## 🧠 一、架构层查漏（致命级，必须补）

### ❌ G1. 事件系统不可靠（致命）

**现状问题：**
```python
def consume():
    return r.xread({STREAM_KEY: "$"}, block=1000)
```

这行代码有 **5 个生产级致命缺陷**：

1. 用 `$` 表示只读新消息，**崩溃重启后丢消息**
2. 没有 Consumer Group，**多 Worker 会重复消费**
3. 没有 ACK 机制，**处理失败消息消失**
4. 没有 PEL（Pending Entry List）清理，**僵尸消息堆积**
5. 没有 Dead Letter Queue，**毒消息会卡死系统**

**✅ 修补方案：**
```python
# 使用 Consumer Group + XAUTOCLAIM
def consume_group(group="waos-workers", consumer="worker-1"):
    # 1. 先认领超时消息（崩溃恢复）
    claimed = r.xautoclaim(STREAM_KEY, group, consumer, 
                           min_idle_time=30000, count=10)
    # 2. 再读新消息
    new_msgs = r.xreadgroup(group, consumer, 
                            {STREAM_KEY: ">"}, count=10, block=1000)
    return claimed[1] + new_msgs[0][1]

def ack(msg_id, group="waos-workers"):
    r.xack(STREAM_KEY, group, msg_id)

def dead_letter(msg, reason):
    r.xadd("waos:dlq", {"original": msg, "reason": reason, 
                         "ts": time.time()})
```

## 🧠 二、业务层查漏（赚钱模块全有问题）

### ❌ B1. Lead 评分模型是玩具级

```python
priority_score = 0.4 * intent + 0.3 * value + 0.2 * stage + 0.1 * persona_fit
```

**致命问题：**
1. 权重是写死的，**无法自适应**
2. 没有**时间衰减**（3天前的评论 vs 刚评论）
3. 没有**负向打分**（用户骂人、广告、已购）
4. 没有**渠道权重**（微信 DM 价值远高于抖音评论）
5. score 没归一化，**intent=100, value=1 时分数爆表**

**✅ 必须升级：**
```python
def calc_priority(lead, persona, ctx):
    # 时间衰减（半衰期 6 小时）
    recency_decay = 0.5 ** (ctx.hours_since_last_action / 6.0)
    
    # 归一化
    intent = sigmoid(lead.intent_score)
    value = min(lead.value_score / 100.0, 1.0)
    
    # 负向惩罚
    penalty = 0
    if lead.is_spam: penalty += 0.5
    if lead.already_customer: penalty += 0.3
    
    # 渠道权重
    channel_w = {"wechat_dm": 1.0, "comment": 0.3, "video": 0.5}
    
    raw = (0.4*intent + 0.3*value + 0.2*lead.stage_score 
           + 0.1*persona.fit_score) * channel_w[lead.source] * recency_decay - penalty
    
    return max(0, min(100, raw * 100))
```

### ❌ B3. 调度器根本不是调度器

```python
if priority > 80: execute_now()
else: queue(lead)
```

这**不是调度**，这是路由。真正的调度器必须解决：

| 问题 | 现状 | 必须补 |
|------|------|--------|
| 公平性 | 无 | 老化机制，COLD 不能饿死 |
| 资源限制 | 无 | Worker 并发上限 |
| 时间窗口 | 无 | 凌晨3点不回微信 |
| 速率限制 | 无 | 单用户每天最多3条 |
| 优先级抢占 | 无 | HOT 进来要踢 COLD |
| 反压 | 无 | 队列满时降级处理 |

**✅ 真正的调度器结构：**
```python
class Scheduler:
    def __init__(self):
        self.hot_q = PriorityQueue()   # ZSET
        self.warm_q = PriorityQueue()
        self.cold_q = PriorityQueue()
        self.aging = AgingPolicy()     # 每10分钟 +5分
        self.cooldown = CooldownPolicy()  # 单用户冷却
        self.business_hours = BusinessHours()
        self.rate_limiter = TokenBucket()
    
    def tick(self):
        if not self.business_hours.is_open():
            return self.defer_to_next_day()
        
        # 老化补偿
        self.aging.apply(self.cold_q)
        
        # 抢占式调度
        while self.worker_pool.has_capacity():
            task = self.hot_q.pop() or self.warm_q.pop() or self.cold_q.pop()
            if not task: break
            if self.cooldown.is_cooled(task.user_id):
                self.worker_pool.submit(task)
            else:
                task.snooze(minutes=30)
```

### Patch 1：可靠事件消费 Worker

```python
# app/event/worker.py
import time
import signal
import redis
from app.event.handler import handle_event
from app.event.dlq import send_to_dlq

r = redis.Redis(...)
STREAM = "waos:event"
GROUP = "waos-workers"
CONSUMER = f"worker-{os.getpid()}"

def init_group():
    try:
        r.xgroup_create(STREAM, GROUP, id="0", mkstream=True)
    except redis.exceptions.ResponseError:
        pass  # 已存在

def run():
    init_group()
    running = True
    def stop(s, f): 
        nonlocal running
        running = False
    signal.signal(signal.SIGTERM, stop)
    
    while running:
        # 1. 认领僵尸消息
        try:
            _, claimed, _ = r.xautoclaim(
                STREAM, GROUP, CONSUMER, 
                min_idle_time=60000, count=10
            )
        except: claimed = []
        
        # 2. 读新消息
        try:
            resp = r.xreadgroup(
                GROUP, CONSUMER,
                {STREAM: ">"},
                count=10, block=1000
            )
            new_msgs = resp[0][1] if resp else []
        except: new_msgs = []
        
        # 3. 处理
        for msg_id, data in claimed + new_msgs:
            try:
                handle_event(data)
                r.xack(STREAM, GROUP, msg_id)
            except Exception as e:
                send_to_dlq(data, str(e))
                r.xack(STREAM, GROUP, msg_id)  # 防止卡死
        
        # 4. 清理 PEL（可选）
        if time.time() % 300 < 1:
            cleanup_pel()
```

### Patch 3：真正的调度器

```python
# app/dispatch/scheduler.py
import time
from dataclasses import dataclass
from typing import Optional

@dataclass
class Task:
    lead_id: str
    priority: float
    enqueued_at: float
    persona_id: Optional[str] = None
    retry_count: int = 0

class Scheduler:
    def __init__(self, redis_client, worker_pool):
        self.r = redis_client
        self.pool = worker_pool
        self.queues = {
            'hot': 'waos:q:hot',
            'warm': 'waos:q:warm',
            'cold': 'waos:q:cold',
        }
        self.cooldown_ttl = 1800  # 30min
        self.business_hours = (9, 22)
    
    def enqueue(self, task: Task):
        q = self._classify(task.priority)
        self.r.zadd(self.queues[q], 
                    {task.lead_id: task.priority})
    
    def tick(self):
        if not self._in_business_hours():
            return
        
        # 老化：COLD 队列每10分钟 +5
        self._apply_aging()
        
        # 调度
        while self.pool.has_capacity():
            task = self._dequeue()
            if not task: break
            
            if self._is_cooled(task.lead_id):
                self.pool.submit(task)
                self._mark_cooled(task.lead_id)
            else:
                # 30分钟后再试
                self.r.zadd(self.queues['warm'],
                            {task.lead_id: task.priority})
    
    def _dequeue(self) -> Optional[Task]:
        for q in ['hot', 'warm', 'cold']:
            items = self.r.zpopmax(self.queues[q], 1)
            if items:
                lead_id, priority = items[0]
                return Task(lead_id, priority, time.time())
        return None
    
    def _apply_aging(self):
        # ZINCRBY 批量加分
        self.r.zincrby(self.queues['cold'], 5, '*')  # 需 Lua 脚本
```

```tsx
export function useKeyboardNav(leads: Lead[], onSelect, onAction) {
  const [cursor, setCursor] = useState(0)
  const [pinned, setPinned] = useState(false)
  
  useEffect(() => {
    const handler = (e: KeyboardEvent) => {
      if (e.target.tagName === 'INPUT') return
      
      switch(e.key) {
        case 'j': setCursor(c => Math.min(c+1, leads.length-1)); break
        case 'k': setCursor(c => Math.max(c-1, 0)); break
        case 'r': onAction('reply', leads[cursor]); break
        case 'e': onAction('escalate', leads[cursor]); break
        case 'h': onAction('human', leads[cursor]); break
        case 'p': setPinned(p => !p); break  // pin
        case '/': e.preventDefault(); openSearch(); break
      }
      onSelect(leads[cursor])
    }
    window.addEventListener('keydown', handler)
    return () => window.removeEventListener('keydown', handler)
  }, [leads, cursor])
  
  return { cursor, pinned }
}
```

#### 3.4 幂等与分布式锁 (`app/core/security.py`)
解决并发竞态和重复消费问题。
```python
import redis
import uuid
from functools import wraps
from fastapi import HTTPException
from app.core.config import settings

r = redis.from_url(settings.REDIS_URL, decode_responses=True)

def acquire_lock(lock_name: str, expire_seconds: int = 10) -> bool:
    """尝试获取分布式锁，防止多Worker并发处理同一用户"""
    lock_id = str(uuid.uuid4())
    acquired = r.set(f"{settings.LOCK_PREFIX}{lock_name}", lock_id, nx=True, ex=expire_seconds)
    return acquired

def release_lock(lock_name: str):
    """释放锁（生产环境应使用 Lua 脚本保证原子性，这里做简化演示）"""
    r.delete(f"{settings.LOCK_PREFIX}{lock_name}")

def idempotent(key_prefix: str, expire_seconds: int = 86400):
    """接口幂等装饰器，防止Webhook重试导致重复建单"""
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, **kwargs):
            # 假设 kwargs 中含有 external_id
            external_id = kwargs.get("external_id")
            if not external_id:
                raise ValueError("Missing external_id for idempotency")
            
            key = f"{settings.IDEM_PREFIX}{key_prefix}:{external_id}"
            
            # SETNX 原子操作
            if not r.set(key, "1", nx=True, ex=expire_seconds):
                raise HTTPException(status_code=409, detail="Duplicate request, already processed")
            
            try:
                return await func(*args, **kwargs)
            except Exception as e:
                # 如果执行失败，回滚幂等键，允许重试
                r.delete(key)
                raise e
        return wrapper
    return decorator
```

#### 3.5 可靠事件总线 (`app/core/event_bus.py`)
解决你原本代码丢消息、崩溃后数据丢失的问题。
```python
import json
import time
import redis
from app.core.config import settings

r = redis.from_url(settings.REDIS_URL, decode_responses=True)

def init_consumer_group():
    """初始化 Consumer Group，防止启动报错"""
    try:
        r.xgroup_create(settings.EVENT_STREAM_KEY, settings.EVENT_GROUP, id="0", mkstream=True)
    except redis.exceptions.ResponseError as e:
        if "BUSYGROUP" not in str(e):
            raise

def publish_event(event_type: str, payload: dict):
    """发布事件到 Stream"""
    event = {
        "type": event_type,
        "payload": json.dumps(payload),
        "ts": time.time()
    }
    r.xadd(settings.EVENT_STREAM_KEY, event)

def consume_events(consumer_name: str, batch_size: int = 10, block_ms: int = 1000):
    """可靠消费事件：先认领僵尸消息，再读新消息"""
    # 1. 认领超时未 ACK 的消息 (崩溃恢复)
    # min_idle_time=60000 表示如果消息停留在 PEL 超过 60s 未 ACK，则被该 consumer 认领
    try:
        claimed_msgs = r.xautoclaim(
            settings.EVENT_STREAM_KEY, 
            settings.EVENT_GROUP, 
            consumer_name, 
            min_idle_time=60000, 
            count=batch_size
        )
        # xautoclaim 返回 (next_start_id, [messages], [deleted_ids])
        if claimed_msgs and len(claimed_msgs) > 1:
            for msg_id, fields in claimed_msgs[1]:
                yield msg_id, fields
    except Exception as e:
        print(f"Autoclaim error: {e}")

    # 2. 读取新消息 (">" 表示从未读过的最新消息开始)
    try:
        resp = r.xreadgroup(
            settings.EVENT_GROUP, 
            consumer_name, 
            {settings.EVENT_STREAM_KEY: ">"}, 
            count=batch_size, 
            block=block_ms
        )
        if resp:
            for _, msgs in resp:
                for msg_id, fields in msgs:
                    yield msg_id, fields
    except Exception as e:
        print(f"Readgroup error: {e}")

def ack_event(msg_id: str):
    """确认消息已处理完毕，从 PEL 中移除"""
    r.xack(settings.EVENT_STREAM_KEY, settings.EVENT_GROUP, msg_id)

def send_to_dlq(msg_id: str, fields: dict, reason: str):
    """毒消息进入死信队列，防止卡死系统"""
    dlq_payload = {
        "original_id": msg_id,
        "fields": json.dumps(fields),
        "reason": reason,
        "ts": time.time()
    }
    r.xadd("waos:dlq", dlq_payload)
    # 就算进了 DLQ，也要 ACK 掉原消息，不然会一直卡在 PEL 里
```

```python
VALID_TRANSITIONS = {

            f"非法状态流转: {current_state} -> {target_state} "

        )
```

### 🧠 2. 真实优先级引擎 (`app/engine/priority.py`)
把玩具级的加法，升级为带**时间衰减、归一化、渠道权重、负向惩罚**的工业级评分。

```python
import math
from datetime import datetime

def sigmoid(x):
    return 1 / (1 + math.exp(-x))

def calc_priority(intent: float, value: float, stage: str, persona_fit: float, 
                  hours_since_last_action: float, source: str, 
                  is_spam: bool = False, already_customer: bool = False) -> float:
    """
    工业级优先级打分算法
    返回 0-100 的分数
    """
    # 1. 时间衰减 (半衰期 6 小时)
    recency_decay = 0.5 ** (hours_since_last_action / 6.0)
    
    # 2. 归一化 (把任意范围的打分压到 0-1)
    norm_intent = sigmoid(intent / 20.0 - 2.5)  # 假设 intent 原始范围 0-100
    norm_value = min(value / 100.0, 1.0)
    
    # 3. 阶段基础分
    stage_scores = {"new": 0.2, "engaged": 0.4, "qualified": 0.6, "hot": 0.9}
    norm_stage = stage_scores.get(stage, 0.1)
    
    # 4. 负向惩罚
    penalty = 0.0
    if is_spam: penalty += 0.5
    if already_customer: penalty += 0.3
    
    # 5. 渠道权重 (微信私域价值远高于公域评论)
    channel_weights = {"wechat_dm": 1.0, "comment": 0.3, "video": 0.5}
    channel_w = channel_weights.get(source, 0.4)
    
    # 6. 综合打分
    raw_score = (
        0.4 * norm_intent + 
        0.3 * norm_value + 
        0.2 * norm_stage + 
        0.1 * persona_fit
    )
    
    final_score = (raw_score * channel_w * recency_decay) - penalty
    
    # 压缩到 0-100 区间
    return max(0.0, min(100.0, final_score * 100))
```

```python
from typing import List

class Persona:
    def __init__(self, id: str, weight: float, capacity: int, active_leads: int, historical_cvr: float):
        self.id = id
        self.weight = weight
        self.capacity = capacity
        self.active_leads = active_leads
        self.historical_cvr = historical_cvr # 历史转化率
        
    def is_available(self) -> bool:
        return self.active_leads < self.capacity

def resolve_conflict(candidates: List[Persona], force_assign: bool = False) -> Persona:
    """
    当多个 Persona 竞争同一个 Lead 时，决定谁接管
    """
    # 1. 过滤掉已满载的 Persona
    available = [p for p in candidates if p.is_available()]
    
    if not available:
        if not force_assign:
            raise OverflowError("所有 Persona 均已满载，进入等待队列")
        # 如果强制分配，选活跃度最低的
        available = candidates
        
    # 2. 按历史转化率降序排
    available.sort(key=lambda p: p.historical_cvr, reverse=True)
    
    # 3. (可选) 这里可以插入 A/B 实验分桶逻辑
    # if lead.in_experiment("exp_001"): return available[1]
    
    # 4. 返回最优 Persona
    return available[0]
```

### 🚀 4. 多级调度器核心 (`app/engine/scheduler.py`)
这是整个系统的心脏。解决任务排队、饿死、抢占、限频问题。

```python
import time
import json
import redis
from datetime import datetime
from app.core.config import settings

r = redis.from_url(settings.REDIS_URL, decode_responses=True)

class Scheduler:
    def __init__(self):
        self.queues = {
            "hot": "waos:q:hot",    # ZSet: lead_id -> priority
            "warm": "waos:q:warm",
            "cold": "waos:q:cold"
        }
        self.cooldown_ttl = 1800  # 单用户冷却 30 分钟
        self.business_hours = (9, 22)  # 9点到22点营业

    def enqueue(self, lead_id: str, priority: float):
        """入队：按分数路由到不同队列"""
        if priority >= 80:
            q = self.queues["hot"]
        elif priority >= 50:
            q = self.queues["warm"]
        else:
            q = self.queues["cold"]
            
        r.zadd(q, {lead_id: priority})

    def _in_business_hours(self) -> bool:
        """时间窗口控制：凌晨不执行任务"""
        hour = datetime.now().hour
        return self.business_hours[0] <= hour < self.business_hours[1]

    def _is_cooled(self, lead_id: str) -> bool:
        """频率控制：防止过度营销封号"""
        return not r.exists(f"waos:cooldown:{lead_id}")

    def _mark_cooled(self, lead_id: str):
        r.setex(f"waos:cooldown:{lead_id}", self.cooldown_ttl, "1")

    def apply_aging(self):
        """
        老化机制：防止 COLD 队列用户被永远饿死
        每次执行，把 COLD 队列里所有人的分数 +2，让其慢慢向上浮
        """
        # 使用 Lua 脚本原子化批量增加 ZSet 分数
        lua_script = """
        local key = KEYS[1]
        local members = redis.call('ZRANGE', key, 0, -1)
        for _, m in ipairs(members) do
            redis.call('ZINCRBY', key, 2, m)
        end
        """
        r.eval(lua_script, 1, self.queues["cold"])

    def dequeue(self, worker_capacity: int) -> dict:
        """
        出队调度：遵循 HOT > WARM > COLD 优先级
        返回任务信息，如果无任务返回 None
        """
        if worker_capacity <= 0:
            return None

        if not self._in_business_hours():
            return None

        # 优先从 HOT 队列拉取
        for q_name in ["hot", "warm", "cold"]:
            q_key = self.queues[q_name]
            
            # ZPOPMAX 取出分数最高的
            items = r.zpopmax(q_key, 1)
            if not items:
                continue
                
            lead_id, priority = items[0]
            
            # 频率控制检查
            if not self._is_cooled(lead_id):
                # 还在冷却期，降级扔回 WARM 队列延迟处理
                r.zadd(self.queues["warm"], {lead_id: float(priority) - 10})
                continue
                
            # 标记进入冷却
            self._mark_cooled(lead_id)
            
            return {
                "lead_id": lead_id,
                "priority": float(priority),
                "queue_type": q_name
            }
            
        return None

    def preempt(self, lead_id: str, current_priority: float):
        """
        抢占机制：当极 HOT 用户进来，如果系统满载，直接把 COLD 用户踢回队列
        """
        # 找到正在被执行的 COLD 任务 (模拟逻辑，需要配合 Worker Pool 状态)
        # 将其从执行中移出，重新入队
        r.zadd(self.queues["cold"], {lead_id: current_priority})
```

### 🧠 2. 全局状态与焦点管理 (`src/store/useOpsStore.ts`)

解决左右联动和新消息打断问题。引入 **“焦点三态”**：
- `FOLLOW`：新 HOT 消息进来自动抢夺右侧焦点（适合巡航模式）
- `PIN`：钉住当前用户，新消息只做红点提示（适合正在深度处理）
- `DND`：免打扰，只入队列不弹窗（适合批量清理）

```typescript
import { create } from 'zustand';

// 焦点三态枚举
export type FocusMode = 'FOLLOW' | 'PIN' | 'DND';

export interface Lead {
  id: string;
  name: string;
  source: 'wechat' | 'comment' | 'video';
  priority: number;
  stage: 'new' | 'engaged' | 'hot' | 'converted';
  last_message: string;
  unread: boolean;
}

interface OpsState {
  leads: Lead[];
  selectedLeadId: string | null;
  focusMode: FocusMode;
  cursor: number; // 键盘流当前索引

  setLeads: (leads: Lead[]) => void;
  addLead: (lead: Lead) => void;
  selectLead: (id: string) => void;
  setFocusMode: (mode: FocusMode) => void;
  moveCursor: (dir: 'up' | 'down') => void;
}

export const useOpsStore = create<OpsState>((set, get) => ({
  leads: [],
  selectedLeadId: null,
  focusMode: 'PIN', // 默认钉住，防止刚进来就乱跳
  cursor: 0,

  setLeads: (leads) => set({ leads }),
  
  addLead: (lead) => {
    const { focusMode, leads, selectedLeadId } = get();
    const newLeads = [lead, ...leads];
    
    set({ leads: newLeads });

    // 焦点管理核心逻辑
    if (focusMode === 'FOLLOW' && lead.stage === 'hot') {
      // 跟随模式：HOT 用户强制抢焦点
      set({ selectedLeadId: lead.id, cursor: 0 });
    } else if (focusMode === 'PIN' && selectedLeadId) {
      // 钉住模式：不动焦点，只加未读红点
      // lead.unread 默认已经是 true
    } else if (!selectedLeadId) {
      // 如果没选过，默认选第一条
      set({ selectedLeadId: lead.id, cursor: 0 });
    }
  },

  selectLead: (id) => {
    const { leads } = get();
    const index = leads.findIndex(l => l.id === id);
    set({ selectedLeadId: id, cursor: index >= 0 ? index : 0 });
  },

  setFocusMode: (mode) => set({ focusMode: mode }),

  moveCursor: (dir) => {
    const { leads, cursor } = get();
    if (dir === 'up' && cursor > 0) {
      const newIndex = cursor - 1;
      set({ cursor: newIndex, selectedLeadId: leads[newIndex].id });
    } else if (dir === 'down' && cursor < leads.length - 1) {
      const newIndex = cursor + 1;
      set({ cursor: newIndex, selectedLeadId: leads[newIndex].id });
    }
  }
}));
```

```python
import logging
import json
import sys
from datetime import datetime
from uuid import uuid4

class StructuredFormatter(logging.Formatter):
    """JSON 格式日志，方便机器解析"""
    def format(self, record):
        log_data = {
            "timestamp": datetime.utcnow().isoformat(),
            "level": record.levelname,
            "message": record.getMessage(),
            "module": record.module,
            "line": record.lineno
        }
        # 把 extra 里的自定义字段带上
        if hasattr(record, 'lead_id'):
            log_data["lead_id"] = record.lead_id
        if hasattr(record, 'trace_id'):
            log_data["trace_id"] = record.trace_id
            
        if record.exc_info:
            log_data["exception"] = self.formatException(record.exc_info)
            
        return json.dumps(log_data)

def setup_logging():
    handler = logging.StreamHandler(sys.stdout)
    handler.setFormatter(StructuredFormatter())
    
    root_logger = logging.getLogger()
    root_logger.setLevel(logging.INFO)
    root_logger.addHandler(handler)
    
    # 降低第三方库噪音
    logging.getLogger("uvicorn.access").setLevel(logging.WARNING)
```

### 🤖 4. 防封号行为模拟器 (`app/engine/human_simulator.py`)
如果你用个人号挂 RPA/Webhook，统一回复频率必封。必须加人类延迟和随机性。

```python
import time
import random
import logging

log = logging.getLogger(__name__)

class HumanSimulator:
    @staticmethod
    def typing_delay(text_length: int) -> float:
        """模拟打字时间：普通人打字速度约 30-40 字/分钟"""
        base_time = text_length / 5.0  # 基础耗时
        # 加入 +-30% 的随机抖动
        jitter = random.uniform(0.7, 1.3)
        delay = base_time * jitter
        # 最少延迟 2 秒，最多不超过 15 秒
        return max(2.0, min(15.0, delay))

    @staticmethod
    def reading_delay() -> float:
        """模拟阅读用户消息的时间"""
        return random.uniform(1.5, 4.0)

    @staticmethod
    def execute_with_human_behavior(action_func, message: str):
        """包裹执行动作，加入人类行为延迟"""
        log.info("Simulating human reading...")
        time.sleep(HumanSimulator.reading_delay())
        
        log.info("Simulating human typing...")
        time.sleep(HumanSimulator.typing_delay(len(message)))
        
        # 真正执行动作 (发微信/发评论)
        action_func(message)
        log.info("Action executed.")
```

### 🧠 5. AI 安全护盾 (`app/engine/safety_shield.py`)
这是用 LLM 做私域的生死线！防止用户通过 Prompt 注入让 AI 发送竞品链接，或乱承诺价格导致纠纷。

```python
import re

class SafetyShield:
    # 1. 敏感词/竞品过滤
    BANNED_KEYWORDS = ["竞品A", "竞品B", "加微信群", "支付宝转账"]
    
    # 2. LLM 绝对不能做的事情 (系统提示词约束)
    SYSTEM_CONSTRAINTS = """
    严格约束：
    1. 绝对不能承诺任何具体价格折扣，只能说“我帮您申请一下优惠”。
    2. 绝对不能提供其他平台的联系方式。
    3. 遇到骂人或政治敏感话题，只回复：“抱歉，无法理解您的意思，请问还有其他产品问题吗？”
    """
    
    @classmethod
    def sanitize_input(cls, user_input: str) -> bool:
        """输入净化：检测 Prompt 注入"""
        # 简单的注入检测
        injection_patterns = [r"ignore.*previous", r"disregard.*instructions", r"system.*prompt"]
        for pattern in injection_patterns:
            if re.search(pattern, user_input, re.IGNORECASE):
                return False # 拦截
        return True
    
    @classmethod
    def filter_output(cls, ai_output: str) -> str:
        """输出过滤：防止 AI 乱说话"""
        for word in cls.BANNED_KEYWORDS:
            if word in ai_output:
                return f"【安全拦截】AI 输出包含违规词: {word}。已替换为兜底话术。"
        
        # 检测承诺价格的违规行为
        if re.search(r"\d+折|便宜\d+元", ai_output):
            return "抱歉，具体优惠我需要帮您向主管申请一下，请稍等。"
            
        return ai_output
```

### 🧠 1. 会话上下文管理器 (`app/engine/context_manager.py`)

解决多轮对话记忆问题。不能每次都把所有历史塞给 LLM，也不能不带记忆。我们用 Redis 做滑动窗口记忆。

```python
import json
import redis
from app.core.config import settings

r = redis.from_url(settings.REDIS_URL, decode_responses=True)

class ContextManager:
    MAX_HISTORY = 10  # 最多保留最近 10 轮对话

    @staticmethod
    def build_context(lead_id: str, user_msg: str, persona_prompt: str) -> list:
        """构建发送给 LLM 的 messages 结构"""
        history_key = f"waos:history:{lead_id}"
        
        # 1. 获取历史记录
        raw_history = r.lrange(history_key, 0, ContextManager.MAX_HISTORY - 1)
        history = [json.loads(h) for h in raw_history]
        
        # 2. 组装 LLM 格式
        messages = [{"role": "system", "content": persona_prompt}]
        messages.extend(history)
        messages.append({"role": "user", "content": user_msg})
        
        return messages

    @staticmethod
    def save_turn(lead_id: str, user_msg: str, ai_reply: str):
        """保存这一轮对话到 Redis"""
        history_key = f"waos:history:{lead_id}"
        # 左插最新的
        r.lpush(history_key, json.dumps({"role": "user", "content": user_msg}))
        r.lpush(history_key, json.dumps({"role": "assistant", "content": ai_reply}))
        # 修剪列表，防止无限增长爆内存
        r.ltrim(history_key, 0, ContextManager.MAX_HISTORY * 2 - 1)
        # 设置 7 天过期
        r.expire(history_key, 86400 * 7)
```

### 🛡️ 2. 带熔断与降级的 LLM 调用器 (`app/engine/llm_executor.py`)

这是生死线。AI 接口超时/限流时，系统不能卡死，必须**熔断**并**降级**到人工或模板兜底。

```python
import logging
import time
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type
from circuitbreaker import circuit
from app.engine.context_manager import ContextManager
from app.engine.safety_shield import SafetyShield

log = logging.getLogger(__name__)

class LLMTimeoutError(Exception): pass

class LLMExecutor:
    def __init__(self, llm_client):
        self.llm = llm_client # 假设传入 OpenAI/通义千问 等客户端

    @circuit(failure_threshold=3, recovery_timeout=30) 
    # 熔断器：连续失败3次，熔断30秒，期间直接走兜底
    @retry(
        stop=stop_after_attempt(2), # 最多重试 2 次
        wait=wait_exponential(multiplier=1, max=3), # 指数退避
        retry=retry_if_exception_type(LLMTimeoutError)
    )
    def _call_llm_api(self, messages: list) -> str:
        """真正调用 LLM API 的方法"""
        try:
            # 伪代码：假设 self.llm.chat.completions.create
            response = self.llm.chat.completions.create(
                model="gpt-4o-mini",
                messages=messages,
                temperature=0.7,
                timeout=5 # 严格限制 5 秒超时
            )
            return response.choices[0].message.content
        except Exception as e:
            log.error(f"LLM API call failed: {e}")
            raise LLMTimeoutError(str(e))

    def generate_reply(self, lead_id: str, user_msg: str, persona_prompt: str) -> str:
        """带完整安全护盾和降级的生成入口"""
        
        # 1. 输入安全净化
        if not SafetyShield.sanitize_input(user_msg):
            return "抱歉，无法理解您的意思，请问还有其他产品问题吗？"

        # 2. 构建上下文
        messages = ContextManager.build_context(lead_id, user_msg, persona_prompt)
        
        try:
            # 3. 调用 LLM (带熔断+重试)
            ai_reply = self._call_llm_api(messages)
            
            # 4. 输出安全过滤
            safe_reply = SafetyShield.filter_output(ai_reply)
            
            # 5. 保存上下文
            ContextManager.save_turn(lead_id, user_msg, safe_reply)
            
            return safe_reply
            
        except Exception as e:
            # 6. 降级策略：LLM 挂了或熔断中，走人工接管或模板兜底
            log.critical(f"LLM fallback triggered for lead {lead_id}: {e}")
            fallback_msg = "【系统兜底】实在抱歉，当前咨询人数较多，您的需求我已记录，主管稍后会亲自为您解答。"
            
            # 触发转人工事件
            from app.core.event_bus import publish_event
            publish_event("dispatch.escalate_human", {
                "lead_id": lead_id,
                "reason": "LLM_FALLBACK"
            })
            
            return fallback_msg
```

```python
            ai_reply = self.llm_executor.generate_reply(lead_id, user_msg, persona_prompt)
            log.info(f"Generated reply: {ai_reply}", extra={"lead_id": lead_id})
            
            # 4. 这里接入真实渠道 SDK 发送消息 (微信/企微)
            # wechat_client.send_msg(lead_id, ai_reply)
            
            # 5. 记录监控指标
            EVENTS_PROCESSED.labels(event_type="reply_sent", status="success").inc()
            
        except Exception as e:
            log.error(f"Task failed: {e}", extra={"lead_id": lead_id})
            EVENTS_PROCESSED.labels(event_type="reply_sent", status="failed").inc()

    def run(self, worker_id: str = "worker-1"):
        """主消费循环"""
        self.is_running = True
        log.info(f"Engine Worker {worker_id} started.")
        
        while self.is_running:
            # 1. 定期执行老化机制 (每 100 次循环执行一次，伪逻辑)
            if int(time.time()) % 600 == 0:
                self.scheduler.apply_aging()
                
            # 2. 假设每个 Worker 当前有 5 个并发容量
            worker_capacity = 5 
            
            # 3. 从多级队列拉取任务
            task = self.scheduler.dequeue(worker_capacity)
            
            if task:
                # 4. 异步执行任务，不阻塞下一次拉取
                thread = threading.Thread(target=self.process_task, args=(task,))
                thread.start()
            else:
                # 无任务时休眠 1 秒，防止 CPU 空转
                time.sleep(1)

# 启动入口
def start_engine(llm_client):
    worker = EngineWorker(llm_client)
    # 在实际项目中，用 uvicorn 的 startup 事件起一个后台线程跑这个 run
    thread = threading.Thread(target=worker.run, args=("worker-1",), daemon=True)
    thread.start()
```

### 🌐 1. 全渠道统一事件适配器 (`app/channels/adapters.py`)

不同平台的 Webhook 报文格式完全不一样，如果像你之前那样直接在入口写 `if event["source"] == "wechat"`，代码会极速膨胀。必须用适配器模式抹平差异。

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass

@dataclass
class UnifiedEvent:
    """系统内部统一的事件结构"""
    external_id: str    # 跨平台唯一ID
    source: str         # wechat / douyin / shipinhao
    user_external_id: str
    msg_type: str       # text / image / comment
    content: str
    raw_payload: dict

class BaseAdapter(ABC):
    @abstractmethod
    def parse(self, raw_payload: dict) -> UnifiedEvent:
        pass

class WechatAdapter(BaseAdapter):
    """微信/企微 Webhook 适配器"""
    def parse(self, raw_payload: dict) -> UnifiedEvent:
        # 假设这是企微回调的加密解密后的明文 JSON
        return UnifiedEvent(
            external_id=f"wx_{raw_payload['MsgId']}",
            source="wechat",
            user_external_id=raw_payload['FromUserName'],
            msg_type=raw_payload['MsgType'],
            content=raw_payload.get('Content', ''),
            raw_payload=raw_payload
        )

class VideoChannelAdapter(BaseAdapter):
    """视频号/抖音评论适配器"""
    def parse(self, raw_payload: dict) -> UnifiedEvent:
        return UnifiedEvent(
            external_id=f"dy_{raw_payload['comment_id']}",
            source="video",
            user_external_id=raw_payload['user']['open_id'],
            msg_type="comment",
            content=raw_payload['text'],
            raw_payload=raw_payload
        )

# 工厂注册中心
ADAPTERS = {
    "wechat": WechatAdapter(),
    "video": VideoChannelAdapter()
}

def parse_event(source: str, payload: dict) -> UnifiedEvent:
    adapter = ADAPTERS.get(source)
    if not adapter:
        raise ValueError(f"Unsupported channel: {source}")
    return adapter.parse(payload)
```

### 🛡️ 2. 防封号执行通道 (`app/channels/safe_dispatcher.py`)

生成回复后，怎么发出去大有学问。秒回、频率太高、文案一模一样必封号。这里整合之前的 `HumanSimulator` 并加入全局频控。

```python
class SafeDispatcher:
    # 渠道频率限制：单设备/单微信号 每分钟最多 3 条
    RATE_LIMITS = {
        "wechat_dm": {"limit": 3, "window": 60},
        "comment": {"limit": 10, "window": 60}
    }

    @staticmethod
    def _check_rate_limit(channel: str, user_external_id: str) -> bool:
        key = f"waos:ratelimit:{channel}:{user_external_id}"
        limit = SafeDispatcher.RATE_LIMITS.get(channel, {})
        max_req = limit.get("limit", 5)
        window = limit.get("window", 60)
        
        current = r.incr(key)
        if current == 1:
            r.expire(key, window)
        return current <= max_req

    @staticmethod
    def _fingerprint_text(text: str) -> str:
        """对相同文案进行微小扰动，防文案查重"""
        # 极简版：在末尾加一个不可见字符或随机空格
        invisible_chars = ["", " ", "  "]
        return text + invisible_chars[hash(text) % 3]

    @staticmethod
    def send_reply(channel: str, user_external_id: str, reply_text: str):
        """安全发送回复"""
        # 1. 频率检查
        if not SafeDispatcher._check_rate_limit(channel, user_external_id):
            log.warning(f"Rate limit exceeded for {user_external_id} on {channel}. Delaying.")
            # 延迟入队，稍后再试 (伪代码)
            return False

        # 2. 文案防指纹
        safe_text = SafeDispatcher._fingerprint_text(reply_text)

        # 3. 人类行为模拟 (阅读 + 打字延迟)
        def actual_send():
            # 这里调用真实的 RPA SDK 或企微 API
            log.info(f"Sending to {user_external_id} via {channel}: {safe_text}")
            pass

        # 在后台线程执行，不阻塞主引擎
        import threading
        thread = threading.Thread(
            target=HumanSimulator.execute_with_human_behavior,
            args=(actual_send, safe_text)
        )
        thread.start()
        return True
```

### 🤝 3. 人工接管工作流 (`app/engine/human_handoff.py`)

AI 不是万能的，遇到高意向客户讨价还价，或者 AI 触发熔断时，必须平滑转交人工。人工处理完后，系统要能接回去继续跑。

```python
import logging
from app.core.database import SessionLocal
from app.models.lead import Lead
from app.engine.state_machine import transition, StateMachineError

log = logging.getLogger(__name__)

class HandoffManager:
    @staticmethod
    def escalate_to_human(lead_id: str, reason: str):
        """AI 挂起任务，转交人工"""
        db = SessionLocal()
        try:
            lead = db.query(Lead).filter(Lead.id == lead_id).first()
            if not lead:
                return
                
            # 1. 状态机流转到阻塞态 (假设有 manual_review 状态)
            # 这里我们复用 blocked 状态表示人工接管中
            try:
                # 合法流转: hot -> blocked (人工介入)
                if lead.stage in ['hot', 'qualified']:
                    lead.stage = transition(lead.stage, 'blocked')
                else:
                    log.warning(f"Cannot escalate lead in {lead.stage} state.")
                    return
                    
                db.commit()
                
                # 2. 发送通知给人工 (飞书/钉钉机器人)
                log.info(f"Lead {lead_id} escalated to human. Reason: {reason}")
                # send_dingding_alert(f"需要人工接管！\n用户: {lead_id}\n原因: {reason}")
                
            except StateMachineError as e:
                log.error(f"Handoff state error: {e}")
                
        finally:
            db.close()

    @staticmethod
    def resume_from_human(lead_id: str, human_reply: str):
        """人工处理完毕，手动触发恢复 AI 模式"""
        db = SessionLocal()
        try:
            lead = db.query(Lead).filter(Lead.id == lead_id).first()
            if not lead or lead.stage != "blocked":
                return
                
            # 1. 状态机回退到 hot
            lead.stage = transition("blocked", "new") # 先回 new
            lead.stage = transition("new", "engaged") # 再到 engaged
            db.commit()
            
            # 2. 把人工的回复记录进上下文，让 AI 知道刚才人工说了啥
            from app.engine.context_manager import ContextManager
            # 伪代码：获取上一条用户消息
            last_user_msg = "用户最后的消息" 
            ContextManager.save_turn(lead_id, last_user_msg, human_reply)
            
            # 3. 重新扔进调度队列，让 AI 接管
            from app.engine.scheduler import Scheduler
            scheduler = Scheduler()
            scheduler.enqueue(str(lead.id), priority=85.0) # 给个高优
            
            log.info(f"Lead {lead_id} resumed by human. AI taking over again.")
            
        finally:
            db.close()
```

### 🧪 1. A/B 实验分桶器 (`app/engine/experiment_manager.py`)

不能让所有流量都走同一个 Persona，必须做分流测试。核心原则：**基于用户 ID 哈希分桶，保证同一用户每次进来都走同一个实验组**。

```python
import hashlib
import logging
from app.core.database import SessionLocal
from app.models.lead import Lead

log = logging.getLogger(__name__)

class ExperimentManager:
    # 实验配置：实验ID -> 组名及权重
    EXPERIMENTS = {
        "persona_strategy_v1": {
            "control": 0.5,    # 50% 流量走保守销售型
            "treatment": 0.5   # 50% 流量走激进逼单型
        }
    }

    @staticmethod
    def _hash_bucket(user_id: str) -> float:
        """将用户ID哈希到 0.0 - 1.0 之间"""
        hash_val = hashlib.md5(user_id.encode()).hexdigest()
        # 取前8位转为整数，再除以最大值得到 0-1 的浮点数
        bucket = int(hash_val[:8], 16) / 0xFFFFFFFF
        return bucket

    @classmethod
    def assign_variant(cls, experiment_id: str, user_id: str) -> str:
        """为用户分配实验组"""
        exp = cls.EXPERIMENTS.get(experiment_id)
        if not exp:
            return "default"

        bucket = cls._hash_bucket(f"{experiment_id}:{user_id}")
        cumulative = 0.0
        
        for variant, weight in exp.items():
            cumulative += weight
            if bucket < cumulative:
                log.info(f"User {user_id} assigned to {variant} for exp {experiment_id}")
                return variant
                
        return list(exp.keys())[-1] # 兜底
```

### 📈 2. 转化漏斗与归因模型 (`app/engine/funnel_tracker.py`)

用户从看视频 -> 评论 -> 私信 -> 成交，跨越了多个节点。必须记录每个节点的的时间戳，成交时才能算清楚是哪个渠道的功劳。

```python
from datetime import datetime
import redis
from app.core.config import settings

r = redis.from_url(settings.REDIS_URL, decode_responses=True)

class FunnelTracker:
    # 漏斗节点定义
    STAGES = ["view", "comment", "dm_engaged", "qualified", "converted"]

    @staticmethod
    def record_event(lead_id: str, stage: str, source: str, value: float = 0.0):
        """记录用户到达某个漏斗节点"""
        if stage not in FunnelTracker.STAGES:
            return

        # 使用 Redis Hash 存储漏斗状态: lead_id -> {stage: timestamp}
        funnel_key = f"waos:funnel:{lead_id}"
        
        # 只有第一次到达该节点才记录 (首次触点归因)
        if not r.hexists(funnel_key, stage):
            r.hset(funnel_key, mapping={
                stage: datetime.utcnow().isoformat(),
                f"{stage}_source": source,
                f"{stage}_value": value
            })
            r.expire(funnel_key, 86400 * 30) # 保留 30 天

    @staticmethod
    def get_attribution(lead_id: str) -> dict:
        """
        获取归因路径：当用户成交时调用，算清楚是谁的功劳
        采用"首次互动"和"最终互动"双重归因模型
        """
        funnel_key = f"waos:funnel:{lead_id}"
        data = r.hgetall(funnel_key)
        
        if not data:
            return {}

        first_touch = None
        last_touch = None
        conversion_value = 0.0

        # 找出首次触点和最终触点
        for stage in FunnelTracker.STAGES[:-1]: # 排除 converted
            if stage in data:
                if not first_touch:
                    first_touch = {"stage": stage, "source": data.get(f"{stage}_source")}
                last_touch = {"stage": stage, "source": data.get(f"{stage}_source")}

        if "converted" in data:
            conversion_value = float(data.get("converted_value", 0.0))

        return {
            "lead_id": lead_id,
            "first_touch": first_touch,
            "last_touch": last_touch,
            "conversion_value": conversion_value,
            "path": {s: data.get(s) for s in FunnelTracker.STAGES if s in data}
        }

# 使用示例：
# 用户评论时: FunnelTracker.record_event(lead_id, "comment", "douyin")
# 用户私信时: FunnelTracker.record_event(lead_id, "dm_engaged", "wechat")
# 用户成交时: FunnelTracker.record_event(lead_id, "converted", "wechat", value=1299.0)
# attribution = FunnelTracker.get_attribution(lead_id) -> 用来生成 CRM 报表
```

### 🏷️ 3. 动态标签引擎 (`app/engine/tag_engine.py`)

静态规则维护成本太高，必须让系统根据用户行为自动打标，并把标签反哺给 `Priority Engine`。

```python
import redis
import re
from app.core.config import settings

r = redis.from_url(settings.REDIS_URL, decode_responses=True)

class TagEngine:
    # 规则字典：关键词 -> 标签
    INTENT_KEYWORDS = {
        "价格|多少钱|怎么收费": "price_sensitive",
        "买|下单|付款": "high_intent",
        "便宜|折扣|优惠": "discount_seeker",
        "怎么用|教程|操作": "product_education"
    }

    @staticmethod
    def auto_tag(lead_id: str, user_message: str):
        """根据用户最新消息自动打标"""
        tags = set()
        
        # 获取已有标签
        existing_tags = r.smembers(f"waos:tags:{lead_id}")
        tags.update(existing_tags)

        # 关键词匹配打标
        for pattern, tag in TagEngine.INTENT_KEYWORDS.items():
            if re.search(pattern, user_message):
                tags.add(tag)

        # 存入 Redis Set (自动去重)
        if tags:
            r.sadd(f"waos:tags:{lead_id}", *tags)
            r.expire(f"waos:tags:{lead_id}", 86400 * 7)

        return list(tags)

    @staticmethod
    def get_tags(lead_id: str) -> list:
        """获取用户所有标签"""
        return list(r.smembers(f"waos:tags:{lead_id}"))

    @staticmethod
    def get_tag_score_modifier(lead_id: str) -> float:
        """
        标签反哺优先级打分：
        high_intent -> +20
        discount_seeker -> +10 (爱占便宜的人容易被转化)
        price_sensitive -> -5 (还在比价，意向稍低)
        """
        tags = r.smembers(f"waos:tags:{lead_id}")
        modifier = 0.0
        
        if "high_intent" in tags: modifier += 20.0
        if "discount_seeker" in tags: modifier += 10.0
        if "price_sensitive" in tags: modifier -= 5.0
        if "churned" in tags: modifier -= 30.0
        
        return modifier
```

### ⎈ 2. K8s 弹性伸缩部署 (`k8s/deployment.yaml`)

真正的生产环境，API 节点和后台 Worker 节点必须分开部署。API 扛 Webhook，Worker 扛计算。并根据 CPU 和 **Redis 队列长度** 自动扩缩容。

```yaml
---
# API 服务部署 (扛并发请求)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: waos-api
spec:
  replicas: 2 # 初始 2 个副本
  selector:
    matchLabels:
      app: waos-api
  template:
    metadata:
      labels:
        app: waos-api
    spec:
      containers:
      - name: waos-api
        image: your-registry/waos-api:latest
        ports:
        - containerPort: 8000
        envFrom:
        - secretRef:
            name: waos-secrets # 从 K8s Secret 注入 DATABASE_URL 等
        # 就绪探针与存活探针 (K8s 流量调度的关键)
        readinessProbe:
          httpGet:
            path: /health
            port: 8000
          initialDelaySeconds: 10
          periodSeconds: 5
        livenessProbe:
          httpGet:
            path: /health
            port: 8000
          initialDelaySeconds: 30
          periodSeconds: 10
        resources:
          requests:
            cpu: "250m"
            memory: "512Mi"
          limits:
            cpu: "1000m"
            memory: "1Gi"

---
# Worker 服务部署 (扛后台计算，不暴露端口)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: waos-worker
spec:
  replicas: 3 # 初始 3 个副本并发消费
  selector:
    matchLabels:
      app: waos-worker
  template:
    metadata:
      labels:
        app: waos-worker
    spec:
      containers:
      - name: waos-worker
        image: your-registry/waos-api:latest # 同一个镜像，改启动命令
        command: ["python", "-m", "app.engine.main_loop"] # 运行 Worker 主循环
        envFrom:
        - secretRef:
            name: waos-secrets

---
# K8s 水平自动扩缩容 (HPA)
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: waos-worker-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: waos-worker
  minReplicas: 3
  maxReplicas: 20 # 大促时最多拉起 20 个 Worker
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70 # CPU 超过 70% 就扩容
  # 甚至可以接入外部指标 (Redis 队列长度) 做自定义扩缩容

  #       name: redis_queue_depth

  #       type: AverageValue
```

### 📺 4. Grafana 监控大屏配置思路

你在第 4 步已经暴露了 Prometheus 指标，现在只需要在 Grafana 配置面板。生产级监控必须包含以下几个核心面板：

1. **API 吞吐量与延迟 (Golden Signals)**
   - QPS (Requests per second)
   - P99 Latency (慢请求排查)
   - HTTP 5xx Error Rate
2. **调度器健康度 (System Heartbeat)**
   - `waos_queue_depth{queue_type="hot"}` (HOT 队列堆积数)
   - `waos_queue_depth{queue_type="warm"}` (WARM 队列堆积数)
   - Worker 消费速率 (Events Processed per sec)
3. **LLM 成本与性能 (AI Core)**
   - LLM 调用成功率 (非熔断状态)
   - LLM P99 响应时间 (是否需要换更快的模型)
   - 兜底触发率 (Fallback Rate，高于 5% 就告警)
4. **业务转化漏斗 (Business Value)**
   - 新增 Lead 数
   - 转化为 HOT 的数量
   - 人工接管率 (Escalation Rate)

**告警配置（Alertmanager）：**
- 如果 `waos_queue_depth` 持续 5 分钟大于 500，发飞书告警："队列大爆发，Worker 不够用了！"
- 如果 `LLM_FALLBACK` 触发率 5 分钟内大于 10%，发飞书告警："AI 接口挂了，全面转人工！"
